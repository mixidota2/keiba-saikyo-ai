# 競馬最強AI - Horse Racing Prediction Agent

## プロジェクト概要

Claude Code上で動作する競馬予想AIエージェントシステム。
Centralized Multi-Agent Architectureにより、回収率の最大化を目指す。

## アーキテクチャ

### Centralized Agent Loop（直列フロー + 末尾批評）

```
User (レース指定)
      │
      ▼
┌─── Plan Agent (中央制御・ループ判断) ◄──────────────┐
│     │                                               │
│     ▼                                               │
│   Collect Agent (情報収集)                           │
│     │                                               │
│     ▼                                               │
│   Reason Agent (推論・スコアリング)                    │
│     │                                               │
│     ▼                                               │
│   Critique Agent (批評) ── verdict ──► Plan Agent ──┘
│                              │          "insufficient" → ループ継続
│                              │          "sufficient"   → ループ終了
│                              ▼
└────────────────────► 最終予想出力
```

**重要**: 各エージェントは直列に実行する。並列実行はしない。
Critique Agentは毎ループの最終ステップとして必ず実行し、
その結果をPlan Agentが受け取ってループ継続/終了を判断する。

### ワークフロー

1. ユーザーがレースを指定（例: 「2026年2月15日 東京11R フェブラリーS」）
2. **Plan Agent** が予想戦略を立案し、収集指示・評価方針を決定
3. **Collect Agent** がWeb検索・過去データから情報を収集
4. **Reason Agent** が収集情報を元にスコアリング・推論
5. **Critique Agent** がループ末尾で推論結果を批評し、verdict（sufficient/insufficient）を出力
6. **Plan Agent** がCritique結果を受け取り判断:
   - `insufficient` → 批評の指摘を反映してStep 3に戻る（最大3ループ）
   - `sufficient` → Step 7へ
7. 最終予想を出力（推奨買い目・期待回収率付き）

## 予想ファクター（重み付け）

### 主要ファクター
- **前走スコア** (重み: 高) - 前走の着順・着差・上がりタイム
- **適性** (重み: 高) - 距離適性・コース適性・馬場適性
- **騎手** (重み: 中) - 騎手の成績・コース相性・乗り替わり効果
- **枠順** (重み: 中) - コース形態に応じた枠の有利不利
- **ローテーション** (重み: 中) - 間隔・前走からの臨戦過程
- **タイム適性** (重み: 中) - 当該コース・距離の基準タイムと各馬の持時計の比較

### 当日補正ファクター（レース直前に適用）
- **馬体重** - 増減と過去の好走体重との比較
- **トラックバイアス** - 当日の馬場傾向（内有利/外有利等）
- **パドック情報** - 可能であれば当日の気配
- **天候・馬場状態** - 良/稍重/重/不良による補正

## データ正確性の絶対ルール（全エージェント共通）

**最重要ルール。このルールはすべてのエージェント・スキルに適用される。**

1. **実在する登録馬のみを使用すること**
   - Web検索で取得した公式出馬表（JRA公式, netkeiba等）に記載されている馬のみを対象とする
   - 出馬表に存在しない馬名を推測・捏造してはならない
   - 馬名・馬番・枠番・騎手名はすべてWeb検索で確認した事実に基づくこと
2. **情報が見つからない場合は「不明」とする**
   - 推測で馬名・成績・オッズ等を埋めてはならない
3. **Critique Agentは登録馬の実在性を最優先でチェックする**
   - 出馬表にない馬が含まれていた場合は即座に severity: "critical" でrejectする

## サブエージェントのツール使用ルール（全エージェント共通）

- **WebSearch等のツールは、独立した検索であれば可能な限り並列に呼び出すこと**
- 例: 各馬の成績検索は互いに独立 → 複数馬を同時に並列検索OK
- 例: オッズ検索とコース傾向検索は独立 → 並列OK
- 検索結果を待ってから次の検索が必要な場合のみ直列にする

## 回収率最大化の方針

1. **オッズとの乖離を狙う**: 実力以上にオッズが高い馬を見つける
2. **期待値ベースの買い目**: 的中率ではなく期待回収率で判断
3. **買い目の絞り込み**: 無駄な買い目を排除し、点数を絞る
4. **券種の最適化**: レース特性に応じて単勝/馬連/三連複等を選択
5. **見送り判断**: 期待値が低いレースは「見送り」を推奨
6. **スコア上位馬の活用**: スコア上位2頭のうち少なくとも1頭は買い目の相手に含める
7. **軸馬の信頼性**: 騎手力ファクターが50点未満の馬を軸馬にしない
8. **ペース整合性**: ペース予想の結論と買い目構成を一致させる

## データ永続化ルール

- すべてのデータは `data/` ディレクトリにJSON形式で保存
- ファイル名規則: `{YYYY-MM-DD}_{会場}_{R番号}_{種別}.json`
- 予想後は必ず `predictions/` に保存し、後で振り返りに使う
- 結果判明後は `results/` に保存し、知見を `knowledge/` に蓄積

## コーディング規約

- データファイルはすべてJSON（UTF-8）
- 日付フォーマット: YYYY-MM-DD
- 会場名: 漢字表記（東京、中山、阪神、京都、小倉、新潟、福島、札幌、函館、中京）
- 馬名: カタカナ表記

## スキル使用方法

- `/predict` - レース予想を実行（メインワークフロー）
- `/collect` - 特定レースの情報収集のみ実行
- `/review` - 過去の予想結果を振り返り、知見を更新
